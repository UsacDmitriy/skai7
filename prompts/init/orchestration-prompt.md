# Orchestration Prompt — координация multi-agent работы над Streamlit-приложением

> Промпт для синхронизации нескольких AI-агентов, параллельно разрабатывающих модули Streamlit MVP: загрузка данных, аналитика, UI-табы, smoke-тесты. Запускается каждые 60-90 минут на хакатоне.

## Когда использовать

- **Data Engineer** строит `data_loader.py` и `data_overview.py`
- **Analytics Engineer** делает `metrics.py` и `charts.py`
- **Streamlit Developer** собирает `app.py` и табы (`tabs/data_tab.py`, `tabs/dashboard_tab.py`, `tabs/details_tab.py`)
- **QA** прогоняет smoke-тесты и готовит демо-скрипт
- Каждый агент **независим**, видит только свой модуль

Без orchestration через 1-2 часа получаете **несовместимые модули**: `data_loader` возвращает словарь с одним набором ключей, `dashboard_tab` ждёт другие; `metrics.py` импортирует функцию, которой нет в `data_loader.py`; табы падают с `KeyError` при запуске `app.py`.

Orchestration промпт — **синхронизирующий слой** между агентами. Запускается Chief Architect-ом с чистым контекстом.

## Зачем нужен отдельный промпт

Любой LLM-агент работает в своей **локальной картине мира**:
- Data Engineer не знает, какие именно метрики нужны Analytics Engineer-у
- Analytics Engineer не видит, какие имена колонок ожидают табы
- Streamlit Developer не знает, что `data_loader` ещё не реализовал кэширование
- QA не в курсе, что сигнатуры функций поменялись час назад

Orchestration prompt **поднимает агента над всеми модулями** и проверяет согласованность стыков. Это не написание кода — это **диагностика архитектурной целостности** того, что собрано разными агентами.

## Как использовать на практике

### Шаг 1. Подготовка контекста (3-5 минут)

Chief Architect собирает в один блок:

```bash
# В терминале проекта:
tree -L 2 app/ data/ output/ tabs/
grep -n "^def " app/*.py app/tabs/*.py 2>/dev/null
grep -n "^import\|^from" app/app.py | head -20
cat app/data_loader.py | head -80
cat AGENTS.md | head -50
# Последние ошибки при запуске:
streamlit run app/app.py 2>&1 | tail -30
```

Заметки от каждого агента: что сделано, что в процессе, какие блокеры.

### Шаг 2. Вставить в LLM (1 минута)

Открыть **отдельный** чат (Cherry Studio / OpenRouter — не тот, в котором работает основной агент, нужен **чистый контекст**).

Скопировать orchestration промпт выше, заполнить секции `[ВСТАВЬТЕ...]`.

Рекомендуемые модели: **Kimi K2.5** (основная), **Qwen** (аналитика/отчёт), **Devstral** (резерв). Задача требует понимания целого — не быстрой реакции.

### Шаг 3. Получить orchestration report (3-5 минут)

LLM выдаст отчёт по формату. Команда **читает вместе** 2-3 минуты.

### Шаг 4. Распределение action items (2 минуты)

Chief Architect озвучивает команде:
- «Data Engineer, тебе P0: исправить ключ track_data → track_points в data_loader.py»
- «Analytics Engineer, тебе P0: переименовать функцию в risk_engine.py»
- «Streamlit Developer, тебе P1: добавить русские метки в risk-таблицу»
- «QA, тебе P1: прогнать smoke-тест после исправлений»

Каждый идёт делать своё.

### Шаг 5. Через 60-90 минут — повторить

Снова собираете контекст, вставляете в orchestration промпт, получаете отчёт. К 17:00 — последний orchestration check, после него только финальная полировка и запись демо-видео.

## Расписание orchestration на день

| Время | Сессия | Что проверяем |
|---|---|---|
| 11:30 | #1 | Каркас собран: data_loader читает CSV, app.py запускается без ошибок, 3 таба в skeleton-виде рендерятся. Проверка что ключи словарей согласованы. |
| 13:00 | #2 | End-to-end сценарий на одном событии: данные загружены → dashboard показывает KPI → details-таб показывает трек и видео-ссылку. |
| 14:30 | #3 | Большая часть кода написана. Проверка acceptance criteria: все 4 KPI, risk-таблица с метками, форма действий, action items по P1/P2. |
| 16:00 | #4 | Полировка. Что недоделано — переводится в P3. Проверка README, проверка оффлайн-режима. |
| 17:00 | #5 | Финальный. Только проверка демо-готовности: запуск → 3 таба → отчёт → actions.csv. Никаких новых фич. |

Пять orchestration сессий × 5 минут = 25 минут на синхронизацию за день. Без неё команда тратит **часы** на разруливание несовместимых модулей в конце дня.

## Что проверяет агент в этом прогоне

1. **Module consistency** — совпадают ли сигнатуры функций между файлами?
   - `data_loader.py` → `load_csv_files()` возвращает `Tuple[dict, dict]`?
   - `metrics.py` → `compute_kpi_metrics(datasets)` принимает тот же словарь, что вернул `load_csv_files`?
   - `tabs/dashboard_tab.py` → вызывает `compute_kpi_metrics` с правильными аргументами?

2. **Import chain** — не нарушена ли цепочка импортов?
   - `app.py` → `tabs.data_tab`, `tabs.dashboard_tab`, `tabs.details_tab` → `data_loader`, `metrics`, `charts`
   - Нет циклических импортов
   - Все импортируемые имена существуют в целевых модулях

3. **Data flow** — проходит ли поток данных без разрывов?
   - CSV-файлы в `data/` → `load_csv_files()` → `datasets` dict → все 3 таба получают данные
   - Ключи словаря, которые возвращает `load_csv_files`, совпадают с теми, что используют табы
   - `st.session_state` или `st.cache_data` используются согласованно

4. **Demo readiness** — можно ли запустить приложение и показать все 3 таба?
   - `streamlit run app/app.py` — без ошибок
   - Переключение между табами — без `KeyError`, `AttributeError`, `ImportError`
   - Форма действий создаёт `output/actions.csv`

## Когда orchestration НЕ нужен

- В первый час работы — модули только закладываются, синхронизировать нечего
- Если команда из 2 человек — синхронизация голосом быстрее
- Если все агенты работают строго последовательно (один модуль за другим)

При параллельной работе 3+ агентов orchestration **критичен** — без него модули разойдутся за 2 часа.

## Альтернатива — orchestration без LLM

Если LLM API недоступен — Chief Architect делает проверку **вручную** по чек-листу:

```
1. Сигнатуры функций в data_loader.py == то, что импортируют табы?
   grep "def " app/data_loader.py → сравнить с grep "from data_loader import" tabs/*.py
2. Ключи словаря datasets совпадают во всех модулях?
   grep "datasets\[" app/tabs/*.py → сверить с return в load_csv_files()
3. Импорты в app.py резолвятся?
   python -c "from app import data_loader, metrics; print('OK')"
4. Smoke-тест: streamlit run app/app.py — все 3 таба без ошибок?
5. README содержит инструкцию по локальному запуску?
6. output/actions.csv создаётся при сохранении формы?
7. Демо-скрипт укладывается в 4 минуты?
```

Если на любой из этих вопросов «нет» — Chief Architect пишет action items команде словами.

## Ловушки, которых избегать

❌ **Запускать orchestration слишком часто** (каждые 30 минут). Команда не успевает писать код.

❌ **Пропускать orchestration вообще.** «Мы и так знаем, кто что делает». Через 2 часа узнаёте, что у Data Engineer ключи `track_data`, а у табов — `track_points`.

❌ **Orchestration в той же чат-сессии, что и основной агент.** Контексты смешиваются, LLM путается. Используйте отдельный чат.

❌ **Игнорировать P0 action items.** «Потом исправим». Через час P0 превратится в неработающее демо.

❌ **Ждать полной готовности всех модулей перед первым orchestration.** Первая сессия в 11:30 — проверить skeleton: загружаются ли CSV, запускается ли app.py, рендерятся ли пустые табы.

❌ **Добавлять новые фичи после 16:00.** После 4-й сессии — только баг-фиксы и полировка.

## Минимальная команда, для которой orchestration окупается

Orchestration окупается с команды от **3 человек**, когда минимум 2 агента пишут код параллельно в разных модулях.

Для команды из 2 человек — синхронизация голосом каждые 30-60 минут эффективнее.

Для команды 4+ человек — orchestration критичен, без него модульный хаос неизбежен.
