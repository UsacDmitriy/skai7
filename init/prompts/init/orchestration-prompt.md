# Orchestration Prompt — координация multi-agent работы

> Промпт для использования когда в команде **несколько AI-агентов работают параллельно** над разными частями MVP, и нужно координировать их результаты. Запускается каждые 30-60 минут на хакатоне для синхронизации.

## Когда использовать

Этот промпт нужен на хакатоне когда:

- **Pilot работает в Cursor Composer** над backend
- **Navigator-2 работает в Continue/Cline** над frontend интеграцией
- **Codex CLI** в фоне делает mock-данные или регенерирует тесты
- **Stitch** генерит варианты UI для нового экрана
- Каждый агент **независим**, видит только свой кусок

Без orchestration через 1-2 часа получаете **несовместимые куски**: бэкенд возвращает один формат, фронт ждёт другой; mock-данные не покрывают edge case из тестов; UI ссылается на endpoint, который ещё не написан.

Orchestration промпт — это **синхронизирующий слой** между агентами. Запускается человеком (обычно Pilot) каждые 30-60 минут с чистым контекстом.

## Зачем нужен отдельный промпт

Любой LLM-агент работает в своей **локальной картине мира**:
- Composer видит только то, что сам сейчас редактирует
- Stitch не знает что в backend
- Codex не знает что в UI
- Continue в боковой панели не помнит, что Composer делал час назад

Orchestration prompt **поднимает агента над всеми частями** и заставляет смотреть на проект целиком. Это не про писание кода — это про **проверку согласованности** того, что уже написано разными агентами.

## Промпт (копируйте целиком)

```
Ты — главный архитектор и интегратор. Команда из 3-4 человек работает
8 часов на хакатоне. У них несколько AI-агентов параллельно делают разные
куски MVP: backend, frontend, mock-данные, UI-генерация. Через 30-60 минут
работы куски расходятся — несовместимые форматы данных, дублирующая логика,
пропущенные интеграции.

Твоя задача: за 5-10 минут проверить состояние проекта и вернуть команде
краткий отчёт с конкретными action items для каждого агента/человека.

Ты НЕ пишешь код в этом промпте. Ты диагностируешь и распределяешь.

ВАЖНЫЕ ПРАВИЛА для тебя:

1. Анализируй ВСЁ что я тебе показываю (файлы, скриншоты, логи).
2. Ищи КОНКРЕТНЫЕ несостыковки между частями. Не "качество кода",
   а "endpoint X возвращает поле foo, а frontend ждёт bar".
3. Каждый action item должен быть РЕАЛИЗУЕМ за 15-30 минут одним агентом.
4. Расставь PRIORITY: P0 (блокирует демо) / P1 (важно к 15:30) /
   P2 (если останется время) / P3 (out of scope сегодня).
5. Если что-то неясно — спрашивай у меня, не выдумывай.

КОНТЕКСТ:

Сейчас [укажи время]: например "13:30, через 5 часов финальный питч".
Команда работает в режиме [Модель A / B / C из playbook/07-team-strategy.md].
Конкретно сейчас [Pilot делает X / Navigator делает Y / третий делает Z].

Цели проекта (из AGENTS.md):

>>>
[ВСТАВЬТЕ сюда AGENTS.md или его ключевые секции:
 цель, acceptance criteria, tech stack]
>>>

Текущее состояние ВСЕХ частей проекта:

>>>
[ВСТАВЬТЕ сюда:
 - Если есть git status / diff — вставьте output
 - Список файлов: tree -L 3 backend/ frontend/ data/
 - Содержимое ключевых файлов: backend/main.py, frontend/App.tsx
 - Что делал Composer в Cursor последний час
 - Что делал Continue/Cline на втором ноутбуке
 - Что Stitch сгенерил последним
 - Backup-видео есть? Питч-черновик есть? Mock golden-данные есть?]
>>>

Известные проблемы / блокеры (если есть):

>>>
[ВСТАВЬТЕ если есть: ошибка X в endpoint Y, UI не рендерится,
 OpenAI квота кончилась и т.п.]
>>>

ТВОЯ ЗАДАЧА — выдать orchestration report по структуре:

---

## STATE OF PROJECT (одной фразой)

Что работает end-to-end, что не работает. Один абзац максимум.

## INCONSISTENCIES (несостыковки между агентами)

Список конкретных проблем в виде:

| ID | Что | Где | Severity | Кто фиксит |
|---|---|---|---|---|
| I-1 | Backend возвращает поле `event_type`, frontend ждёт `type` | backend/main.py:45 ↔ frontend/IncidentList.tsx:23 | P0 | Pilot |
| I-2 | Mock-данные имеют 5 записей, тесты ожидают 10 | data/incidents.json ↔ scripts/smoke.py | P1 | Navigator-1 |

Если несостыковок нет — напиши "OK, all parts are consistent".

## ACCEPTANCE CRITERIA STATUS

Из AGENTS.md acceptance criteria. Для каждого:
- ✅ ЗАКРЫТО (с указанием файла где)
- ⚠️ ЧАСТИЧНО (что осталось)
- ❌ НЕ НАЧАТО

Это самое важное. Если 2+ critical критериев ❌ — команда в зоне риска.

## RISK ASSESSMENT

Топ-3 риска для финального демо. Конкретные:
- "Если фронтенд не подключится к API к 15:00 — нет live-демо"
- "Если OpenAI квота кончится в 16:00 — все AI-функции встанут"

Что делать чтобы каждый риск снизить.

## ACTION ITEMS BY ROLE

Конкретно каждому участнику команды на следующие 30-60 минут:

**Pilot (в Cursor):**
- [P0] Что сделать конкретно
- [P1] Что после
- [P2] Что если успеет

**Navigator-1 (продакт):**
- [P0] Что сделать
- ...

**Navigator-2 (UI/дизайн):**
- [P0] Что сделать
- ...

**(4-й, если есть):**
- [P0] Что сделать

## NEXT ORCHESTRATION CHECK

Через сколько минут запустить меня снова. Обычно 30-60 минут.

## QUESTIONS FOR ME (вопросы команде)

Если что-то в твоём анализе зависит от ответа команды — спрашиваешь.
Например: "Решено ли уже, какая модель LLM используется в production?"
Если вопросов нет — пиши "None, proceed with action items".

---

ПОСЛЕ output ЖДИ ответы команды на твои вопросы.
НЕ начинай сам писать код.
НЕ предлагай переписать архитектуру с нуля.

Помни: команда УЖЕ работает 1-2-3 часа. Их задача —
закрыть acceptance criteria к 18:00. Твоя задача —
помочь им не разойтись от своих агентов.

ВЫДАЙ orchestration report.
```

## Как использовать на практике

### Шаг 1. Подготовка контекста (3-5 минут)

Pilot собирает в один блок:

```bash
# В терминале проекта:
git status
git log --oneline -10
tree -L 3 backend/ frontend/ data/
cat AGENTS.md
cat backend/main.py | head -100
cat frontend/src/App.tsx | head -50
```

Скриншот Stitch если был генерирован недавно. Заметки от Navigators о том, что они делали.

### Шаг 2. Вставить в LLM (1 минута)

Открыть **отдельный** Cursor Chat или Claude.ai (не тот, в котором работает Composer — нужен **чистый контекст**).

Скопировать orchestration промпт выше, заполнить секции `[ВСТАВЬТЕ...]`.

Лучшая модель для orchestration: **Claude Opus 4.7** (если VPN), **Qwen Max** или **Kimi K2** (без VPN). Эта задача требует понимания целого, не быстрой реакции — берите медленную и умную.

### Шаг 3. Получить orchestration report (3-5 минут ожидания)

LLM выдаст отчёт по формату. Команда **читает вместе** в течение 2-3 минут.

### Шаг 4. Распределение action items (2 минуты)

Pilot читает action items вслух команде:
- «Pilot, тебе P0: исправить поле `event_type` в backend»
- «Navigator-1, тебе P0: добавить 5 mock-инцидентов»
- «Navigator-2, тебе P0: подключить второй экран Stitch»

Каждый идёт делать своё.

### Шаг 5. Через 30-60 минут — повторить

Снова собираете контекст, снова вставляете в orchestration промпт, снова получаете отчёт. К 16:30 (за 1.5 часа до питча) — последний orchestration check, после него только полировка.

## Когда orchestration НЕ нужен

- В первый час работы — нечего синхронизировать, агенты только начали
- В Modeле A («один pilot») — все знают, что pilot делает, синхронизация не нужна
- Если команда из 2 человек — синк голосом быстрее

В Моделях B и C orchestration **критичен** — без него агенты разойдутся за 2 часа.

## Расписание orchestration на день

Если используете эту практику:

| Время | Что |
|---|---|
| 11:30 | Первый orchestration. Каркас собран, агенты только разошлись по задачам. Проверка что они согласованы. |
| 13:00 | Второй (после обеда). Проверка что end-to-end сценарий работает на cherry-picked примере. |
| 14:30 | Третий. Большая часть кода написана, проверка acceptance criteria. |
| 16:00 | Четвёртый. Полировка. Что недоделано — переводится в P3. |
| 17:00 | Пятый. Финальный. Только на проверку готовности к питчу. |

Пять orchestration sessions × 5 минут = 25 минут на orchestration за день. Это окупается многократно — без него команда тратит **часы** на разруливание несостыковок в конце дня.

## Альтернатива — orchestration без LLM

Если LLM API не работает в моменте — orchestration делается **вручную** Pilot'ом с чек-листом:

```
1. Endpoint'ы в backend/main.py == то, что ждёт frontend?
2. Mock-данные покрывают golden examples?
3. Smoke-тест проходит?
4. Все P0 acceptance criteria закрыты?
5. У всех агентов одна и та же версия AGENTS.md?
6. Есть ли что-то, что агенты делают ДВА раза параллельно?
7. Backup-видео записано (если время после 16:30)?
```

Если на любой из этих вопросов «нет» — Pilot пишет action items команде словами.

## Ловушки которых избегать

❌ **Запускать orchestration слишком часто** (каждые 15 минут). Команда не успевает работать — постоянно прерывается на чтение отчётов.

❌ **Пропускать orchestration вообще.** «Мы всё знаем». Через 3 часа узнаёте, что один агент строил под REST, другой под GraphQL.

❌ **Orchestration в той же чат-сессии что Composer.** Контексты смешиваются, LLM путается. Используйте отдельный чат.

❌ **Игнорировать P0 action items.** «Это потом». Через час P0 превратится в неработающее демо.

❌ **Делать orchestration ДО первого end-to-end прогона.** Нечего ещё orchestrать, агенты не успели нагенерить. Первая сессия — не раньше 11:30.

## Минимальная команда, для которой orchestration окупается

Orchestration окупается с команды от **3 человек с 2+ ноутбуками**, когда минимум 2 агента работают параллельно.

Для команды из 2 человек — синхронизация голосом каждые 30 минут эффективнее.

Для команды 4+ человек — orchestration критичен, без него хаос.
